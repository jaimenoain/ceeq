## Supabase/Resend Auth Loop – Complete

- **Supabase client utilities:** `lib/supabase/client.ts` (browser, `createBrowserClient`) and `lib/supabase/server.ts` (server, `createServerClient` with Next.js `cookies()`) use `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`. Middleware (`middleware.ts`) refreshes the auth session on each request and writes cookies to the response.
- **Server Actions:** `app/actions/auth.ts` implements login, signup, forgot password (reset email), and logout. All use the server Supabase client and redirect or return errors. Input validation matches DOMAIN_MODEL: `email` (valid email), `password` (min 8 characters, message: "Password must be at least 8 characters").
- **Login page:** `/login` uses Shadcn Card, Tabs (Log in / Sign up / Forgot password), Input, Button, and inline labels. Forms are wired to the auth Server Actions; errors and success (forgot password) display inline. The `(auth)` layout remains minimal.
- **Protected routes:** Unauthenticated users hitting `/` are redirected to `/login`. Dashboard routes under `(dashboard)` are protected in `app/(dashboard)/layout.tsx`: `supabase.auth.getUser()` is awaited; if no user, redirect to `/login`. Authenticated users can access the dashboard; the sidebar Logout button submits a form that calls `logoutAction`, which signs out and redirects to `/login`.
- **Tenant isolation:** RLS and tenant-scoped access rely on `auth.jwt().tenant_id` in the database. Ensure the Supabase project sets a custom claim `tenant_id` (e.g. via Auth hooks or `app_metadata`) when users sign up or are assigned to a tenant so RLS policies can enforce isolation.
- **Resend (email provider):** To use Resend for auth emails (signup confirmation, password reset), configure Supabase to use a custom SMTP provider: In the **Supabase Dashboard** go to **Project → Authentication → Providers → Email**, then under **SMTP Settings** (or **Custom SMTP**) set the SMTP server to Resend (e.g. `smtp.resend.com`, port 465/587) and use your Resend API key as the password. Enable the email provider and, if desired, disable "Confirm email" for development.

## Tenant Administrative Foundation – Complete

Administrative foundation for the Tenant: searchers can manage account settings; analysts are locked out of the configuration area.

### Route Security

- **`/settings` is secured:** `app/(dashboard)/settings/layout.tsx` runs before the settings page. If `session?.user?.app_metadata?.role === 'analyst'`, the request is intercepted and the user is redirected to `/pipeline`. Only searchers (or users without role set) can access `/settings`.

### Database & RLS

- **Tenants UPDATE policy:** `supabase/migrations/0001_tenants_update_rls_searcher.sql` adds an RLS policy on `public.tenants` for UPDATE:
  - `USING (id = (auth.jwt() ->> 'tenant_id')::uuid)`
  - `WITH CHECK ((auth.jwt() ->> 'role') = 'searcher')`
  Analysts have no UPDATE policy on `tenants`, so they cannot change tenant profile from the database layer.

### Backend

- **Server Action:** `app/actions/tenant.ts` implements `updateTenantAction(tenantId, name)` (invoked via FormData). Input validation in `lib/validations/tenant.ts`: `tenantId` must be a valid UUID, `name` must be at least 2 characters (matching UX_SPEC Zod schema; no Zod package installed). On success, `revalidatePath("/settings")` and `revalidatePath("/")` are called so the dashboard layout and settings page reflect the new tenant name.

### Frontend

- **Settings UI:** `app/(dashboard)/settings/page.tsx` (server) fetches the current tenant using `user.app_metadata.tenant_id` and passes it to `SettingsContent`. Page title: "Account Settings". `SettingsNavTabs` (Shadcn Tabs) map to URL search params: `?tab=profile`, `?tab=team`, `?tab=fields` (Profile, Team, Custom Fields). `ProfileTabPanel` contains a form with:
  - Tenant Name input mapped to `TenantDTO.name`
  - Read-only utilized seats display mapped to `TenantDTO.max_seats`
  - Submit triggers `updateTenantAction`
- **Optimistic UI:** `useOptimistic` and `useTransition` are used so that when the tenant name is successfully changed, the form shows the new value immediately and, after revalidation, the global app header (sidebar tenant name in `app/(dashboard)/layout.tsx`) reflects the new name. No React Query; layout refetches tenant after `revalidatePath("/")`.
